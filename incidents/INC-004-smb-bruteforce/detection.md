# Detection – INC-004 SMB Brute Force

## 1. Basic Kibana Filter (Lab)

Data view: `winlogbeat-*`

Search bar:

`event.code: "4625" AND host.name: "DESKTOP-DPU3CDQ" AND winlog.event_data.TargetUserName: "administrator"`

Time range: focus on the attack window (for example, Last 15 minutes) to see the spike generated by the `netexec` run against the Windows host `DESKTOP-DPU3CDQ` (172.16.0.10).

## 2. Brute Force Pattern (Elastic Query – Lab Example)

Goal: detect many failed logons on the same account in a short time window from a single source.

Once you know your Kali IP as it appears in the events (field `winlog.event_data.IpAddress`), use:

`event.code: "4625" AND host.name: "DESKTOP-DPU3CDQ" AND winlog.event_data.TargetUserName: "administrator" AND winlog.event_data.IpAddress: "172.16.0.11"`

Then, in Kibana, use a bucket aggregation on:

- `@timestamp` (5–10 minute window)
- optionally `winlog.event_data.IpAddress` to see which sources are generating the most failures

You are looking for:

- High count of 4625 events  
- Same account (`administrator`)  
- Same source IP (the Kali attacker in the lab)

## 3. Detection Rule Concept

High-level rule logic:

- Condition:
  - Count of Event ID 4625 where:
    - `host.name = "DESKTOP-DPU3CDQ"`
    - `TargetUserName = "administrator"`
    - `winlog.event_data.IpAddress = "172.16.0.11"`
  - Threshold example: more than 20 failures in 5 minutes

- Severity:
  - **High** when the account is privileged (local admin, domain admin, service account)

- Mapping:
  - MITRE ATT&CK: **T1110 – Brute Force**

This rule would alert on activity similar to your `netexec` SMB attack and any future brute-force or password-spraying attempts against administrator accounts in this lab.
